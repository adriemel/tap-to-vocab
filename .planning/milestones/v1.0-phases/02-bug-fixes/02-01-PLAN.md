<task type="auto">
  <name>Task 2: BUG-02 — iOS/Safari voice loading</name>
  <files>tap-to-vocab/assets/js/tapvocab.js</files>
  <action>
Replace the voice loading block at lines 27-32 with a synchronous-first pattern that also keeps the async event listener:

```javascript
// BUG-02: check synchronously first (Chrome/Firefox have voices immediately),
// then listen for async load (required on iOS/Safari where voiceschanged may
// fire before this script runs or not at all).
if (typeof speechSynthesis !== "undefined") {
  var _initialVoices = speechSynthesis.getVoices();
  if (_initialVoices.length) {
    _voicesLoaded = true;
    getSpanishVoice();
  }
  if (speechSynthesis.onvoiceschanged !== undefined) {
    speechSynthesis.addEventListener("voiceschanged", function () {
      _voicesLoaded = true;
      getSpanishVoice();
    });
  }
}
```

Do NOT remove the `u.lang = "es-ES"` assignment inside `speakSpanish`. The lang attribute is the graceful fallback when no Spanish voice object is found — it requests Spanish phonetics from the TTS engine even without an explicit voice. Do not add a polling loop or setInterval.
  </action>
  <verify>
Open tapvocab.js and confirm the synchronous `speechSynthesis.getVoices()` call exists before the `voiceschanged` listener, wrapped in `if (typeof speechSynthesis !== "undefined")`. Browser test on Chrome: load topic.html?cat=Animales, tap a word — audio plays. Manual test note for iOS: tapping a word on iOS Safari must produce audio (not silence) even if the voiceschanged event fired early.
  </verify>
  <done>Voice loading uses synchronous check first, then async listener. speakSpanish never silently fails — it speaks with whatever voice is available or falls back to browser default with u.lang="es-ES" set.</done>
</task>
